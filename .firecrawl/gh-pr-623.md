## Summary

- **Fused dense CQ4 MLP kernel** (`cactus_dense_mlp_cq_fused`) that collapses the 5-step `gate -> GeLU(tanh) -> up -> multiply -> down` chain into two thread-pool dispatches instead of five barriers. Refactors the M==1 path of `cactus_quant_matmul` into three reusable static helpers (prepare-activations, build-expanded, gemv-int8-block-range) so the new kernel and the existing matmul share one source of truth. Wired through the graph layer (new `OpType::DENSE_MLP_CQ_FUSED` appended to keep serialized OpType values stable, builder method, dispatch entry, compute node) and substituted into `Gemma4Model::build_mlp` automatically when gate/up/down are CQ4 with matching `group_size`.
- **Synthetic 30-layer Gemma-4 E2B decode benchmark** in `cactus-graph/tests/test_nn.cpp run_benchmarks()` issuing one transformer block per layer (rms\_norm, fp16 Q/K/V/O matmuls, `attention_cached` with kv-cache, MLP variant, residuals) at realistic shapes (hidden=2304, d\_ffn=9216, num\_heads=8, num\_kv\_heads=2, head\_dim=256, gs=128, kv-cache window=256). Reports per-step ms and tok/s for fused vs unfused.
- **Two pre-existing profile-path bugs in the graph executor** uncovered while wiring up that bench: `op_type_names[]` was missing ~10 entries and had ABS/POW out of order (profile output mislabeled ops); the profile loop dispatched INPUT nodes and threw "Unknown operation type: 0". Neither affects the default non-profile path.

## Numbers (M4 Pro, 8 perf cores)

Single dense MLP at gemma-4-E2B FFN shape (M=1, hidden=2304, d\_ffn=9216, gs=128):

|  | ms |
| --- | --- |
| unfused | ~0.66 |
| fused | ~0.55 |
| **delta** | **-15 to -18%** |

Synthetic 30-layer decode (run-to-run jitter is wide because we're shaving ~2 ms out of a ~50 ms step):

|  | ms/step | tok/s |
| --- | --- | --- |
| unfused | ~50 | ~20 |
| fused | ~47 | ~21 |
| **delta** | -2 to -4 ms | +4 to +16% |

End-to-end gain is smaller than the kernel-level -15% because the FFN chain is only ~1/3 of the per-step cost in this synthetic; fp16 Q/K/V/O attention projections dominate alongside MLP. Real-model decode with NPU prefill / fused QKV / GQA shape would shift this fraction, but the kernel win is what it is.

Both correctness tests pass: `Dense MLP CQ4 fused vs unfused` (kernel-level: max-abs diff < 1e-3 vs the canonical `gelu+matmul+matmul+multiply+matmul` chain) and `Dense MLP CQ4 fused (graph)` (same comparison via the graph builder + execute path). Activations are prepared twice in the fused kernel (once each for gate / up Hadamard params) since CQ4 weights may carry different `input_scale` / signs / permutation across matrices; the fusion win is from collapsing the four inter-op barriers, not from sharing activation prep.

```
Ports the gemma4-dense-mlp-fused work from main onto V2's CQ4
(Trellis-Quant 4-bit) path. Collapses the 5-step gate -> GeLU(tanh)
-> up -> multiply -> down chain into a single graph op that uses
two thread-pool dispatches (one per phase) instead of five barriers.

Refactor (cactus-kernels/src/matmul.cpp): extract the M==1 path of
cactus_quant_matmul into three reusable static helpers --
cactus_quant_prepare_activations_internal (Hadamard + per-group int8
quantize), cactus_quant_build_expanded_internal (codebook-index ->
int8 N-block-of-4 expansion, no-op when W->expanded is cached at load
time), and cactus_quant_gemv_int8_block_range (the int8xint8 dotproduct
GEMV restricted to an N-block range). cactus_quant_matmul now delegates
to these; output is bit-equivalent to the previous body.

New kernel cactus_dense_mlp_cq_fused builds on those helpers:
prepares h_in activations twice (once per gate / up Hadamard params --
they may differ across weight matrices in V2), then runs phase 1
worker tiles that each do gate gemv -> gelu -> up gemv -> multiply
on a 32-N-block (128-element) tile with the shared h_in_int8 inputs,
writing the resulting tile of h_full directly. After phase 1, h_full
is prepared with down_W's params, then phase 2 worker tiles run the
down gemv. Workers steal tiles via an atomic counter and the calling
thread participates, matching the dispatch shape of the source branch.

Graph wiring: OpType::DENSE_MLP_CQ_FUSED appended to the enum (so
existing serialized OpType values are unchanged), builder method
dense_mlp_cq_fused, dispatch entry, and compute_dense_mlp_cq_fused_node
in ops_nn.cpp. The model side (Gemma4Model::build_mlp) detects CQ4
gate/up/down with matching group_size and substitutes the fused op;
otherwise the canonical chain runs unchanged.

Tests (cactus-graph/tests/test_nn.cpp):
* Dense MLP CQ4 fused vs unfused -- builds three CQ4 weight matrices
  and asserts the fused output matches the explicit gelu/matmul/
  multiply/matmul chain to within 1e-3 max-abs (effectively tight,
  since both paths flow through the same kernel primitives).
* Dense MLP CQ4 fused (graph) -- same comparison via the graph
  builder + execute path.
* Dense-MLP fused vs unfused benchmark in run_benchmarks at
  Gemma-4-E2B-like dims (M=1, hidden=2304, d_ffn=9216, gs=128).

On M4 Pro (8 perf cores), the bench reports:
  unfused 1x2304x9216  0.697 ms
  fused   1x2304x9216  0.568 ms   (-18.5%)

with both correctness tests passing. All pre-existing
cactus-kernels and cactus-graph tests continue to pass and
cactus-engine builds cleanly with the build_mlp change.

Signed-off-by: Noah Cylich <noahcylich@gmail.com>
```

```
Adds a perf bench inside cactus-graph/tests/test_nn.cpp run_benchmarks()
that issues a 30-layer transformer block per iteration with realistic
shapes (hidden=2304, d_ffn=9216, num_heads=8, num_kv_heads=2,
head_dim=256, gs=128, kv-cache window=256) and reports per-step ms +
tok/s for the fused vs unfused MLP variants. Each layer is rms_norm ->
fp16 Q/K/V matmuls -> attention_cached -> O proj -> rms_norm ->
residual -> rms_norm -> {dense_mlp_cq_fused | gelu+matmul+matmul+
multiply+matmul} -> rms_norm -> residual.

Weights are shared across the 30 layers (one CQ4 triple, ~30 MB) to
keep memory tractable; the graph still issues 30 distinct MLP op
nodes. The KV cache is pre-warmed to current_seq_len = window each
iteration so attention runs at full window every step.

The CQ4 helper struct attach_cq4 plumbs the preexpanded weight layout
into the BufferDesc::cq_expanded / cq_norm_f32 unique_ptr fields,
mirroring what cactus-graph/src/io.cpp does on real model load.
Without this, every CQ matmul rebuilds the int8 expansion from packed
indices on each call, which dominates step time.

On M4 Pro (8 perf cores), the bench reports approximately:
  unfused (30L) ~45 ms  ~22 tok/s
  fused   (30L) ~41 ms  ~25 tok/s   (median +4-5% tok/s)

Per-op profile of one unfused step: MATMUL ~85% (fp16 QKVO + CQ4
gate/up/down), ATTENTION_CACHED ~5%, GELU + others <2%. The fused
MLP saves ~2 ms out of ~43 ms, so the end-to-end delta is smaller
than the kernel-level -18% on the FFN chain alone.

Signed-off-by: Noah Cylich <noahcylich@gmail.com>
```
