## Summary

- Send single-token CQ matmul to the existing packed GEMV kernels when weights are mmap-backed and do not have an attached expanded layout.
- Report chat throughput as decode throughput after the first token, so the displayed tok/s is not diluted by TTFT/prefill time.

## Why

On `v2`, Gemma 4 E2B CQ4 decode uses mmap-backed CQ weights without attached `expanded` / `norm_f32` buffers. The current `M == 1` dispatch only uses the packed GEMV kernels for some group-size cases; otherwise it can continue into the generic CQ matmul path. For the Gemma 4 CQ4 weights, routing single-token decode directly to the packed GEMV kernel avoids that slower generic path while keeping the packed mmap weight representation unchanged.

## Verification

- `source ./venv/bin/activate && cactus build`
- `/tmp/e2e_cactus_decode_v2 --model-path weights/gemma-4-e2b-it --model-name gemma4-cq4-final --prompt 'Write a compact technical explanation of speculative decoding in one paragraph.' --max-tokens 96 --warmup 1 --reps 3`
  - decode: 25.00, 25.16, 25.09 tok/s
- `cmake --build cactus-kernels/build --target test_matmul -j 8 && ./cactus-kernels/build/test_matmul`
  - all 5 tests passed
- `./cactus-engine/tests/build/chat weights/gemma-4-e2b-it --prompt 'Write a compact technical explanation of speculative decoding in one paragraph.'`
  - displayed decode throughput: 24.3 tok/s
  - reported RAM: 569.9 MB
- Fresh interactive process at idle after model load, measured externally with `ps` / `vmmap -summary`:

  - RSS: 1,566,992 KB
  - physical footprint: 491.1 MB
  - mapped file residency: 1.0 GB of 2.0 GB mapped
